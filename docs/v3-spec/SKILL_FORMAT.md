# Catpilot Skill Format v3.0 — Specification

**Status:** Draft for review
**Audience:** Engineers authoring or consuming Catpilot security skills
**Compatibility:** Anthropic Agent Skill superset

---

## 1. Goals

A Catpilot skill is a single self-contained Markdown file that:

1. Loads as a valid **Anthropic Agent Skill** in any runtime that already supports them (Claude Code, etc.) with no Catpilot-specific tooling.
2. Carries optional **`catpilot:`** metadata that Catpilot-aware tooling can consume for severity scoring, control mapping, and validation.
3. Runs **locally with zero phone-home**. The OSS distribution does not transmit telemetry, crash reports, or skill execution events anywhere.
4. Is reviewable by a skeptical security engineer in under five minutes.

A skill is **content**, not a runtime. The agent runtime executes the rules described in the body. Catpilot does not ship a sandbox, interpreter, or daemon as part of the OSS skill set.

---

## 2. File Conventions

### 2.1 Extension

All skills use the `.skill.md` double extension.

```
secret-blocking.skill.md
cloud-cli-safety.skill.md
sql-injection-prevention.skill.md
```

The `.skill.md` extension makes skills greppable, distinguishable from generic README content, and unambiguous to runtime loaders. Plain `.md` files are not treated as skills.

### 2.2 Naming

- **kebab-case**, lowercase ASCII only.
- Filename SHOULD match the `id` frontmatter field exactly (minus the extension).
- No spaces, no underscores, no Unicode.

```
✅ secret-blocking.skill.md
✅ cloud-cli-safety.skill.md
✅ k8s-rbac-review.skill.md

❌ Secret_Blocking.skill.md
❌ secret blocking.skill.md
❌ secret-blocking.md
```

### 2.3 Directory Layout

```
your-repo/
├── skills/
│   ├── source/                       # AUTHORING granularity (per concern)
│   │   ├── core/
│   │   │   ├── secret-blocking.skill.md
│   │   │   └── cloud-cli-safety.skill.md
│   │   ├── frameworks/
│   │   │   ├── django.skill.md
│   │   │   └── nextjs.skill.md
│   │   └── advanced/
│   │       └── agent-identity-integrity.skill.md
│   └── dist/                         # SHIPPING granularity (per tier) — generated
│       ├── catpilot-security-core.skill.md
│       ├── catpilot-django-security.skill.md
│       └── catpilot-security-advanced.skill.md
└── docs/
    └── v3-spec/
        ├── SKILL_FORMAT.md
        └── PACKAGING.md
```

**Source skills** (`skills/source/...`) are authored per concern. They are the unit of diffs, reviews, and per-component versioning.

**Bundle skills** (`skills/dist/...`) are generated by the bundler from source skills. They are what end users install. See §7 (Bundles) and [`PACKAGING.md`](./PACKAGING.md).

A runtime SHOULD load all `*.skill.md` files under `skills/dist/` recursively. Source skills are not loaded directly by runtimes — they're build inputs.

### 2.4 Per-Runtime Installation

| Runtime | How skills are loaded | Notes |
|---|---|---|
| Claude Code | `~/.claude/skills/` or project `.claude/skills/` | Native Anthropic Agent Skill loader. `.skill.md` works as-is. |
| Cursor | `.cursor/rules/` (linked or copied) | Cursor reads `*.mdc` / `*.md`; symlink `.skill.md` files. |
| OpenClaw | `skills/` in workspace, or `~/.openclaw/skills/` | Loaded via skill registry. |
| Cline | `.clinerules/` | Symlink or copy. |
| Aider | `.aider.conf.yml` `read:` list | Reference each skill explicitly. |
| Codex CLI | `--instructions "$(cat skills/*.skill.md)"` | No native skill loader; pass via flag. |
| GitHub Copilot | `.github/copilot-instructions.md` aggregation | Copilot reads a single file; concatenate skills at install time. |

The Catpilot installer (out of scope for this spec) is responsible for creating the appropriate symlinks, config entries, or aggregated files. Skills themselves are runtime-agnostic.

## 3. Frontmatter Schema

Frontmatter is **YAML**, delimited by `---` lines, and is **required**.

### 3.1 Pure-Anthropic fields (required)

These fields MUST be present and MUST conform to the Anthropic Agent Skill schema. A skill missing any of these is malformed.

```yaml
---
name: Secret Blocking
description: |
  Detects and blocks hardcoded secrets (API keys, tokens, private keys,
  database connection strings) before they reach git history. Covers 40+
  patterns across major cloud and SaaS providers.
---
```

| Field | Type | Required | Notes |
|---|---|---|---|
| `name` | string | yes | Human-readable. Title Case. |
| `description` | string (multiline OK) | yes | One-paragraph what + when. Used by agent runtimes for skill selection. Keep under ~500 chars. |

> **Forward-compat note:** Anthropic's Agent Skill schema is evolving. Catpilot tracks the public spec and adds fields to this "pure-Anthropic" set as they stabilize. Authors MAY include any other field defined by Anthropic; Catpilot validators will pass them through unchanged.

### 3.2 Catpilot extension block (optional but recommended)

All Catpilot-specific metadata lives under a single `catpilot:` key. This isolates extensions from the Anthropic surface and makes future schema migrations clean.

```yaml
catpilot:
  id: secret-blocking
  version: 1.0.0
  severity: critical
  category: secrets
  applies_to:
    languages: [any]
    frameworks: [any]
    runtimes: [claude-code, cursor, openclaw, cline, aider, copilot]
  control_mappings:
    soc2: [CC6.1, CC6.6, CC7.2]
    pci_dss: ["3.4", "3.5", "8.2.1"]
    iso_27001: [A.10.1.1, A.9.4.3]
    nist_csf: [PR.DS-1, PR.DS-5, PR.AC-1]
    owasp_top_10: [A02:2021, A07:2021]
  provenance:
    origin: catpilot
    incident_derived: false
  references:
    - https://owasp.org/www-project-top-ten/
  maintainers:
    - team: catpilot-security
```

| Field | Type | Required | Values / Notes |
|---|---|---|---|
| `catpilot.id` | string | yes | kebab-case, matches filename (minus `.skill.md`). Stable identifier. |
| `catpilot.version` | semver string | yes | `MAJOR.MINOR.PATCH`. Bump MAJOR on breaking rule changes. |
| `catpilot.severity` | enum | yes | `critical \| high \| medium \| low \| info` |
| `catpilot.category` | string | yes | One of: `secrets`, `cloud-cli`, `injection`, `crypto`, `auth`, `supply-chain`, `ci-cd`, `iac`, `data-handling`, `agent-safety`, `incident-response`, `general`. |
| `catpilot.applies_to.languages` | string[] | no | e.g. `[python, typescript]` or `[any]`. |
| `catpilot.applies_to.frameworks` | string[] | no | e.g. `[django, fastapi]` or `[any]`. |
| `catpilot.applies_to.runtimes` | string[] | no | Subset of supported runtimes. Empty/omit = all. |
| `catpilot.control_mappings` | object | no | Keys: `soc2`, `pci_dss`, `iso_27001`, `nist_csf`, `owasp_top_10`. Values are string arrays of control IDs. |
| `catpilot.provenance.origin` | enum | no | `catpilot \| community \| upstream`. |
| `catpilot.provenance.incident_derived` | bool | no | `true` if the skill exists because of a real incident. Sets the tone of the body. |
| `catpilot.provenance.incident_link` | url | no | Optional postmortem URL. |
| `catpilot.references` | url[] | no | Public references (OWASP, CVE, vendor docs). |
| `catpilot.maintainers` | object[] | no | `{ team: ..., handle: ... }`. |
| `catpilot.deprecated` | bool | no | `true` if superseded. Pair with `catpilot.replaced_by`. |
| `catpilot.replaced_by` | string | no | `id` of replacement skill. |

### 3.3 Severity scale

| Severity | Definition |
|---|---|
| `critical` | Active risk of data loss, credential exposure, or production outage. Block by default. Examples: hardcoded secrets, partial-YAML cloud updates, `rm -rf /`. |
| `high` | Likely to introduce vulnerabilities or compliance failures. Strongly warn; require explicit confirmation. Examples: SQL injection patterns, missing CSRF tokens, force-push to protected branches. |
| `medium` | Bad practice that increases attack surface. Warn. Examples: floating Docker tags, missing request timeouts, overly permissive CORS. |
| `low` | Style-adjacent or low-blast-radius. Surface as a comment. Examples: missing rate limit headers, verbose error messages in dev. |
| `info` | Informational guidance, no enforcement. Examples: "consider using parameterized queries when possible". |

### 3.4 Compatibility matrix

| Field | Anthropic-native | Catpilot extension | Notes |
|---|---|---|---|
| `name` | ✅ | — | |
| `description` | ✅ | — | |
| `model` (if/when Anthropic adds) | ✅ | — | Pass-through. |
| `tools` (if/when Anthropic adds) | ✅ | — | Pass-through. |
| `catpilot.*` | — | ✅ | Ignored by non-Catpilot runtimes. |

A non-Catpilot runtime that ignores unknown YAML keys (Anthropic's loader does) will load Catpilot skills as plain Anthropic Agent Skills with no degradation.

## 4. Body Conventions

The body is the part the agent actually reads at runtime. Optimize it for an LLM consuming a few thousand tokens of guidance, not a human reading a wiki.

### 4.1 Required sections

Every skill body MUST contain these sections, in this order, as `##` headings:

1. **`## When to apply`** — one paragraph. What triggers this skill.
2. **`## Rules`** — the actual normative content. Use ❌ / ✅ pairs and severity tags.
3. **`## Remediation`** — concrete fix patterns. Real code, not pseudocode.

### 4.2 Recommended sections

- **`## Why`** — context, especially for `incident_derived: true` skills. Lead with the incident narrative.
- **`## Negative examples`** — what malicious or accidental misuse looks like. Helps the agent recognize it.
- **`## Detection patterns`** — regex or AST patterns the agent can use to scan code.
- **`## References`** — links to upstream docs, CVEs, postmortems.

### 4.3 Severity tags inline

Inside the body, mark individual rules with bracketed severity tags so an agent can prioritize when context is tight:

```markdown
- **[critical]** Never commit `AKIA*` strings to source control.
- **[high]** Avoid `password=` literals in connection strings.
- **[medium]** Prefer secret managers over `.env` for production deploys.
```

Inline tags MUST match the `catpilot.severity` field's allowed values.

### 4.4 Code blocks

- Always specify a language: ```` ```python ````, ```` ```bash ````, ```` ```yaml ````, ```` ```hcl ````.
- Mark dangerous examples with a comment: `# ❌ DANGEROUS:` or `// ❌ NEVER`.
- Mark safe examples: `# ✅ SAFE:` or `// ✅ CORRECT`.
- Keep blocks under ~25 lines. Long blocks hurt agent comprehension.

### 4.5 Agent-runtime hints

Optional `> [!agent]` blockquote callouts give runtime-specific instructions. These are advisory, not enforced.

```markdown
> [!agent] When you detect any pattern in the "Detection patterns" section,
> stop, surface the match to the user, and require explicit confirmation
> before continuing. Do not auto-remediate.
```

## 5. Spec Examples (snippets)

### 5.1 Minimal valid skill

```markdown
---
name: No Force Push To Main
description: Block git push --force or --force-with-lease to main/master/production branches.
catpilot:
  id: no-force-push-main
  version: 1.0.0
  severity: high
  category: general
---

## When to apply

Any time the agent is about to invoke `git push` on a branch named
`main`, `master`, `production`, or matching `release/*`.

## Rules

- **[high]** Never run `git push --force` or `git push -f` on a protected branch.
- **[high]** `--force-with-lease` is also blocked on protected branches.
- **[medium]** Prefer `git revert` over rewriting history on shared branches.

## Remediation

```bash
# ❌ DANGEROUS
git push --force origin main

# ✅ SAFE: push to a feature branch and open a PR
git checkout -b fix/regression
git push origin fix/regression
```
```

### 5.2 Frontmatter with full Catpilot extension

```yaml
---
name: Cloud CLI Safety
description: Require query-before-modify, full-command display, and rollback prep for any Azure/AWS/GCP/k8s/terraform CLI invocation that mutates infrastructure.
catpilot:
  id: cloud-cli-safety
  version: 1.0.0
  severity: critical
  category: cloud-cli
  applies_to:
    languages: [any]
    frameworks: [any]
    runtimes: [claude-code, cursor, openclaw, cline, aider, copilot]
  control_mappings:
    soc2: [CC6.1, CC7.2, CC8.1]
    pci_dss: ["6.4.5", "10.2"]
    iso_27001: [A.12.1.2, A.12.5.1]
    nist_csf: [PR.IP-1, PR.IP-3, DE.CM-7]
  provenance:
    origin: catpilot
    incident_derived: true
  references:
    - https://learn.microsoft.com/en-us/azure/container-apps/azure-resource-manager-api-spec
---
```

## 6. Validation Rules

A skill is **valid** if and only if all of the following hold. Catpilot ships a validator (out of scope for this spec) that checks each rule.

### 6.1 Structural

1. Filename ends in `.skill.md`.
2. Filename is kebab-case ASCII.
3. File begins with `---\n`, contains exactly one closing `---` for frontmatter.
4. Frontmatter parses as valid YAML.
5. Body is non-empty after frontmatter.

### 6.2 Required fields

6. `name` is a non-empty string.
7. `description` is a non-empty string.
8. If `catpilot:` is present:
   - `catpilot.id` matches `^[a-z][a-z0-9-]*$` and equals the filename minus `.skill.md`.
   - `catpilot.version` matches semver `^\d+\.\d+\.\d+$`.
   - `catpilot.severity` ∈ {critical, high, medium, low, info}.
   - `catpilot.category` ∈ the allowed category enum.

### 6.3 Body

9. Body contains `## When to apply`, `## Rules`, `## Remediation` headings (case-sensitive).
10. Every fenced code block has a language tag.
11. Inline severity tags (`[critical]`, `[high]`, etc.) only use allowed enum values.

### 6.4 Cross-field

12. If `catpilot.deprecated: true`, `catpilot.replaced_by` is set and points to an existing skill `id` in the same repo.
13. If `catpilot.provenance.incident_derived: true`, the body SHOULD contain a `## Why` section.

### 6.5 Soft warnings (not failures)

- Body length > 8 KB → warn (consider splitting).
- More than 20 inline severity tags of the same level → warn (audit for noise).
- Missing `control_mappings` for a `severity: critical` skill → warn.

### 6.6 Malformed examples

```yaml
# ❌ Wrong: missing description
---
name: Foo
catpilot: { id: foo, version: 1.0.0, severity: high, category: general }
---

# ❌ Wrong: id does not match filename "secret-blocking.skill.md"
---
name: Secret Blocking
description: ...
catpilot:
  id: secrets   # mismatch
  version: 1.0.0
  severity: critical
  category: secrets
---

# ❌ Wrong: severity not in enum
catpilot:
  severity: blocker  # invalid
```

## 7. Bundles

A **bundle** is a single shipped skill whose body is the concatenation of multiple component skills. Bundles let v3.0 separate **authoring granularity** (per-concern source files for clean diffs and independent versioning) from **install granularity** (one skill per tier so users don't drown in install decisions).

See [`PACKAGING.md`](./PACKAGING.md) for the strategy and tier definitions. This section specifies the file-level shape only.

### 7.1 Bundle frontmatter

A bundle is a valid Catpilot skill that additionally declares `catpilot.bundle`:

```yaml
---
name: Catpilot Security Core
description: |
  Baseline security guardrails covering secrets, cloud CLI, local CLI, database,
  Docker, secrets management, PII, supply chain, and language-baseline patterns.
  Use whenever generating, reviewing, or modifying code.
catpilot:
  id: catpilot-security-core
  version: 3.0.0
  severity: critical          # = max(component.severity)
  category: general           # bundles use `general` unless single-category
  bundle:
    is_bundle: true
    components:
      - id: secret-blocking
        version: 3.0.0
        source: skills/source/core/secret-blocking.skill.md
      - id: cloud-cli-safety
        version: 3.0.0
        source: skills/source/core/cloud-cli-safety.skill.md
      # …
  control_mappings:           # = union(component.control_mappings)
    soc2: [CC6.1, CC6.6, CC7.2, CC8.1]
    pci_dss: ["3.4", "3.5", "6.4.5", "8.2.1", "10.2"]
    iso_27001: [A.9.4.3, A.10.1.1, A.12.1.2, A.12.5.1]
    nist_csf: [PR.AC-1, PR.DS-1, PR.DS-5, PR.IP-1, PR.IP-3, DE.CM-7]
    owasp_top_10: [A02:2021, A07:2021]
---
```

| Field | Type | Required | Notes |
|---|---|---|---|
| `catpilot.bundle.is_bundle` | bool | yes for bundles | Always `true` if present. Source skills omit `catpilot.bundle` entirely. |
| `catpilot.bundle.components[].id` | string | yes | Must equal the source skill's `catpilot.id`. |
| `catpilot.bundle.components[].version` | semver | yes | Must equal the source skill's `catpilot.version` at build time. |
| `catpilot.bundle.components[].source` | path | yes | Repo-relative path to the source skill. Used by the bundler and by validators. |
| `catpilot.bundle.depends_on` | string[] | no | Other bundle `id`s that must also be installed (e.g. framework extensions list `catpilot-security-core`). |

### 7.2 Body shape

The bundle body is the concatenation of component bodies, each prefixed with a stable level-2 heading derived from the component `id`:

```markdown
## When to apply

The bundle's overall trigger guidance. Each component below has its own,
more specific, application criteria.

## Rules

### secret-blocking

<body of secret-blocking.skill.md, with its top-level `## When to apply`,
`## Rules`, `## Remediation` collapsed to `### Trigger`, `### Rules`,
`### Remediation` so the bundle's outer headings stay clean>

### cloud-cli-safety

<…>

## Remediation

See per-component remediation under `## Rules` above.
```

The bundler MUST emit deterministic output: stable component order (taken from `catpilot.bundle.components[]`), stable heading slugs, no timestamp comments. Two builds from the same source tree produce byte-identical bundle files.

### 7.3 Severity and control-mapping aggregation

- Bundle `catpilot.severity` = max severity across components (critical > high > medium > low > info).
- Bundle `catpilot.control_mappings.<framework>` = sorted union of components' values for that framework.
- Bundle `catpilot.applies_to.languages` / `.frameworks` = intersection if all components specify; `[any]` otherwise.

These rules are mechanical. The bundler computes them; authors do not hand-edit.

### 7.4 Versioning

- Each component is versioned independently in its source file (`catpilot.version`).
- The bundle has its own `catpilot.version`.
- Bundle MAJOR bumps when any component's MAJOR bumps, or when components are added/removed.
- Bundle MINOR bumps on component MINOR bumps.
- Bundle PATCH bumps on component PATCH bumps or bundler output changes.

This lets `secret-blocking@3.1.0` ship without forcing every other component to bump, while still giving end users a single bundle version to pin.

### 7.5 Validation additions

In addition to §6:

14. A bundle's `catpilot.bundle.components[].id` MUST match an existing source skill's `catpilot.id`.
15. A bundle's `catpilot.bundle.components[].version` MUST equal the source skill's `catpilot.version` at the build commit.
16. A bundle's `catpilot.severity` MUST equal the max of its components' severities.
17. A source skill (one without `catpilot.bundle`) MUST NOT live under `skills/dist/`. Bundle files MUST live under `skills/dist/`.
18. A source skill MUST live under `skills/source/<tier>/...` where `<tier>` is one of `core`, `frameworks`, `advanced`.

### 7.6 Non-bundle skills

A skill without `catpilot.bundle` is a normal source skill. Everything in §2–§6 applies unchanged. Source skills are the unit of authoring; bundles are the unit of shipping. Most contributors will only ever touch source skills.

## 8. Forward Compatibility

### 7.1 Tracking Anthropic schema changes

Anthropic's Agent Skill format is the upstream. When Anthropic adds, renames, or removes a frontmatter field:

1. Catpilot follows within one minor release.
2. Newly required Anthropic fields are added to §3.1 with a deprecation window for skills missing them.
3. Catpilot extension fields are never given names that collide with the Anthropic surface. All Catpilot extensions live under the `catpilot:` key.

### 7.2 Versioning the spec itself

This document is versioned as part of the Catpilot guardrails repo. Spec changes follow semver:

- **MAJOR**: breaking changes to required fields, body section names, file extension, or validation rules.
- **MINOR**: new optional fields, new categories, new control-mapping frameworks.
- **PATCH**: clarifications, typo fixes, examples.

Skills declare compatibility via `catpilot.version`, not via the spec version. The validator infers required spec version from the fields used.

### 7.3 Migration path

When the spec version increments MAJOR, Catpilot ships:

1. A migration note in `docs/v3-spec/MIGRATIONS.md` (future).
2. A `catpilot-skills migrate` validator subcommand that rewrites old skills to the new schema where mechanical, and flags non-mechanical changes for human review.
3. A grace period of at least one minor release where both schemas validate.

## 9. Non-Goals

Explicitly out of scope for this spec:

- **Runtime execution semantics.** How Claude Code, Cursor, OpenClaw, etc. choose to apply a skill is up to that runtime.
- **Telemetry.** OSS skills do not phone home. Anything observability-related is the runtime's concern, not the skill's.
- **Dynamic skills.** This spec covers static, file-based skills only. Dynamic skills (fed by a security event collector under commercial agreement) are a separate Catpilot SaaS surface and are not described here.
- **Skill marketplaces.** Discovery, signing, and distribution of skills are addressed in a separate document.
