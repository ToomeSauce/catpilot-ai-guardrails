# Skill Format

**Status:** Active.
**Conformance:** Catpilot skills are valid [Anthropic Agent Skills](https://agentskills.io/specification). The `metadata.catpilot.*` block is a Catpilot extension that other runtimes ignore.
**Distribution:** Skills install via [`npx skills add`](https://github.com/vercel-labs/skills) and appear on the [skills.sh](https://skills.sh) leaderboard.

---

## 1. Why this format

Catpilot ships security skills into 51+ AI coding agents (Claude Code, Cursor, Codex, OpenClaw, Cline, Aider, GitHub Copilot, OpenCode, …). Every one of them expects the same on-disk shape:

```
skills/
  <skill-name>/
    SKILL.md
    [scripts/]
    [references/]
    [assets/]
```

We use that shape exactly. We do not invent a new file extension, layout, or installer. Catpilot's value is in the *content* of the skills (real-incident-derived security guidance) and the *bundling* of related skills into install-friendly tiers — not in re-shaping the ecosystem.

## 2. Layout

### 2.1 A single skill

A skill is a directory whose name matches its `name` field:

```
skills/secret-blocking/
├── SKILL.md          # required: metadata + instructions
├── references/       # optional: long-form material loaded on demand
│   └── REFERENCE.md
├── scripts/          # optional: executable helpers
│   └── scan.py
└── assets/           # optional: templates, schemas, fixtures
```

The directory name **must** equal the `name` field in the frontmatter. The agent CLI walks `skills/` and treats every directory containing a `SKILL.md` as an installable skill.

### 2.2 Source vs. shipped

Catpilot maintains skills in two places in this repo:

| Path | Purpose | Visible to `npx skills add`? |
|---|---|---|
| `src/skills/<tier>/<name>/SKILL.md` | **Source** — what authors edit. One source skill per security concern. Never installed directly. | No (outside `skills/`). |
| `skills/<bundle-name>/SKILL.md` | **Shipped bundles** — generated by the bundler from the source skills. This is what end users install. | Yes. |

This split means authors get fine-grained version control and review per concern, while end users see a small number of installable bundles. See [`PACKAGING.md`](./PACKAGING.md) for the bundling strategy.

### 2.3 Tiers

Source skills are organised by tier under `src/skills/`:

```
src/skills/
├── core/                         → packs into skills/catpilot-security-core/
├── frameworks/
│   ├── django/                   → packs into skills/catpilot-django-security/
│   ├── fastapi/                  → packs into skills/catpilot-fastapi-security/
│   ├── rails/                    → packs into skills/catpilot-rails-security/
│   ├── express/                  → packs into skills/catpilot-express-security/
│   ├── nextjs/                   → packs into skills/catpilot-nextjs-security/
│   ├── springboot/               → packs into skills/catpilot-springboot-security/
│   └── docker/                   → packs into skills/catpilot-docker-security/
└── advanced/                     → packs into skills/catpilot-security-advanced/
```

Tier names are stable; framework names match what the bundler auto-detects in user projects.

## 3. SKILL.md frontmatter

Every `SKILL.md` starts with YAML frontmatter that follows the Anthropic Agent Skills spec exactly:

```yaml
---
name: secret-blocking
description: One- or two-sentence summary of what this skill does and when to use it. Max 1024 chars. Include keywords agents will match against.
license: MIT
compatibility: Optional. Use only when the skill needs specific tools or runtime versions.
metadata:
  catpilot:
    id: secret-blocking
    version: 1.0.0
    severity: critical
    category: secrets
    applies_to:
      languages: [any]
      frameworks: [any]
      runtimes: [claude-code, cursor, openclaw, cline, aider, github-copilot, codex]
    control_mappings:
      soc2: [CC6.1, CC6.6, CC7.2]
      pci_dss: ["3.4", "3.5", "3.6", "8.2.1"]
      iso_27001: [A.9.4.3, A.10.1.1, A.10.1.2]
      nist_csf: [PR.AC-1, PR.DS-1, PR.DS-5]
      owasp_top_10: ["A02:2021", "A07:2021"]
    provenance:
      origin: catpilot
      incident_derived: false
    maintainers:
      - team: catpilot-security
    references:
      - https://owasp.org/www-project-top-ten/A02_2021-Cryptographic_Failures/
      - https://docs.github.com/en/code-security/secret-scanning/about-secret-scanning
---
```

### 3.1 Spec fields (required + optional)

These are defined by the Agent Skills spec. Catpilot does not redefine them.

| Field | Required | Constraints |
|---|---|---|
| `name` | yes | 1–64 chars, lowercase a–z + digits + hyphens, must match parent directory name, no leading/trailing/consecutive hyphens |
| `description` | yes | 1–1024 chars, single line recommended, agent-discoverable keywords |
| `license` | no, but Catpilot **requires** it | License name (`MIT`, `Apache-2.0`) or reference to a bundled `LICENSE` file |
| `compatibility` | no | Free-form environment requirement note. ≤500 chars. |
| `metadata` | no, but Catpilot uses it | Free-form key-value map. Catpilot extensions live under `metadata.catpilot.*`. |
| `allowed-tools` | no | Experimental Anthropic field. Not used by Catpilot today. |

### 3.2 The `metadata.catpilot.*` extension

Other agents ignore unknown metadata keys, so the Catpilot block is invisible to them and load-bearing for our tooling.

| Key | Required | Type | Notes |
|---|---|---|---|
| `id` | yes | string | Stable identifier. Equals `name`. Used by the bundler to deduplicate. |
| `version` | yes | semver string | Per-skill version. Independent of bundle version. |
| `severity` | yes | enum | `critical \| high \| medium \| low \| info`. Drives bundler aggregation rules. |
| `category` | yes | string | Free-form taxonomy (`secrets`, `cloud-cli`, `db-safety`, `agent-identity`, …). |
| `applies_to.languages` | no | string list | `[any]` for universal skills. |
| `applies_to.frameworks` | no | string list | `[any]` for universal skills. |
| `applies_to.runtimes` | no | string list | Targeted agent runtimes. Empty/omitted = all. |
| `control_mappings.<framework>` | no | string list | Compliance control IDs. Frameworks: `soc2`, `pci_dss`, `iso_27001`, `nist_csf`, `owasp_top_10`. |
| `provenance.origin` | no | string | `catpilot`, `community`, `vendor:<name>`, etc. |
| `provenance.incident_derived` | no | bool | True when the skill exists because of a real production incident. |
| `maintainers` | no | object list | `[{team: catpilot-security}]` style entries. |
| `references` | no | URL list | External authoritative references for the skill. |

The bundler reads `severity`, `version`, `applies_to`, and `control_mappings` to compose bundle frontmatter. See `PACKAGING.md` §3.

### 3.3 Severity scale

| Level | Meaning | Bundler treatment |
|---|---|---|
| `critical` | Will cause data loss, credential leakage, or production outage. Always-on. | Bundle severity becomes `critical` if any component is. |
| `high` | Likely to cause real harm in common workflows. | Strongly recommended. |
| `medium` | Best-practice. Catches subtle bugs and policy violations. | Recommended. |
| `low` | Style and consistency. | Optional. |
| `info` | Informational only. Never produces a halt. | Optional. |

## 4. SKILL.md body

The Anthropic spec places **no format requirements** on the body. Write whatever helps an agent perform the task.

Catpilot conventions (recommended, not validator-enforced):

| Section | Purpose |
|---|---|
| `## When to apply` | Trigger conditions. Be concrete. |
| `## Why` | Threat model in plain language. Cite an incident if `provenance.incident_derived: true`. |
| `## Rules` | Bulleted, severity-tagged. `**[critical]**` etc. |
| `## Detection patterns` | If applicable, regex / heuristic table. |
| `## Negative examples` | Code blocks that should fail. |
| `## Remediation` | Code blocks that should pass. Multiple patterns when relevant. |
| `## References` | Links to standards, vendor docs, OWASP, internal incident reports. |

Keep `SKILL.md` under 500 lines / ~5000 tokens. Push detailed material into `references/REFERENCE.md` (or topic-specific files) and link with relative paths. Agents load reference files on demand, which preserves context budget.

## 5. Optional companion directories

| Dir | Use for |
|---|---|
| `references/` | Long-form supplementary docs, regex tables, full incident write-ups, compliance crosswalks. |
| `scripts/` | Self-contained helpers an agent may execute (`scan.py`, `validate.sh`). Document dependencies. Handle errors gracefully. |
| `assets/` | Templates, schemas, fixtures, JSON Schema files, example configs. |

Keep references one level deep. Avoid nested chains of cross-references.

## 6. Validation rules

A `SKILL.md` is valid when **all** of the following hold:

1. YAML frontmatter parses.
2. `name` and `description` are present and within length limits.
3. `name` matches `^[a-z0-9](?:[a-z0-9]|-(?!-))*[a-z0-9]$` (Anthropic spec) and matches the parent directory name.
4. `description` is ≤1024 chars.
5. `license` is set (Catpilot requirement, not Anthropic).
6. `metadata.catpilot.id` equals `name`.
7. `metadata.catpilot.version` is a valid semver string.
8. `metadata.catpilot.severity` is one of the enum values in §3.3.
9. `metadata.catpilot.category` is set.
10. Body is non-empty after the frontmatter.
11. Body length is ≤500 lines (warning, not error, above 500).

The validator (PR #3, follow-up) lives at `tools/validate-skill.py` and runs in CI on every PR.

## 7. Versioning

| Change | Bump |
|---|---|
| Wording fix, link update, formatting | patch |
| New rule, new pattern, expanded coverage, new framework support | minor |
| Severity change, ID rename, removed rule, breaking metadata change | major |

Bundles version independently from their components. See `PACKAGING.md` §3.

## 8. Forward compatibility

- New optional `metadata.catpilot.*` keys may be added in any minor release without bumping major.
- Removing or renaming an existing `metadata.catpilot.*` key is a major bump for that skill.
- Anthropic spec additions (new top-level fields) are absorbed transparently.

## 9. Non-goals

- A skill is not a runtime program. There is no execution model defined here. Runtimes decide how to surface skill text to their agents.
- A skill does not phone home. The OSS distribution has zero telemetry. Skills run entirely from the local file system.
- A skill is not a replacement for a static analyzer. It guides agent behavior at code-generation time. Pair with traditional SAST/DAST/SCA in CI for defense in depth.
- A skill does not authenticate the user. Identity, RBAC, and audit live in the SaaS layer (out of scope here).
